## 3. 飞行阶段（状态机）与各阶段运动性质

每艘船按 `stage`（配合 `direction`）走一个状态机。一次完整任务（母站→目标→母站）的阶段序列与运动如下表。**「这一程」= 一次单向 `stage 0` 巡航 + 其后的入坞动画**。

| stage | 我命名 | 去程 dir=+1 的运动 | 回程 dir=−1 的运动 | 运动类型 | 闭式解 |
|---|---|---|---|---|---|
| ≤ −2 | **停机盘滑行** | `t:0→1` @0.03335，贴在出发站停机盘，→ −1 | `t:1→0` @0.03335，到 t<0 卸货回收 | 固定时长（线性 t） | ✅ 30 帧 |
| −1 | **起降过渡** | `t:0→1` @num14，沿盘法线 smoothstep 抬升至 25m，→ 0 | `t:1→0` @num14·⅔，从 `pPosTemp` 降到盘上，→ −2 | 固定时长（位置 smoothstep，**速度=0**） | ✅ 1/num14 或 1.5/num14 帧 |
| | | ⚠️ 注意：抬升的是**位置**（径向 25m），但 `uRot=reference.uRot*shipDiskRot` 使船头朝向**切向**；进入 stage0 时 `uVel=uRot.forward` 即**切向初速**，非"垂直起飞"。详见 **§8**。 | | | |
| 0 | **巡航（主）** | 加速→巡航→（曲速）→减速，直到 `num21 < 6·num10` → stage=+1 | 同左 → stage=−1 | **变速直线/曲线**，见 §4 | 分段闭式（本文重点） |
| 1 | **入坞过渡** | `t:1→0` @num14·⅔，降入目标坞，→ 2 | `t:0→1` @num14，从坞抬升，→ 0 | 固定时长（smoothstep，速度=0） | ✅ 1.5/num14 或 1/num14 帧 |
| ≥ 2 | **装卸** | `t:1→0` @0.0334，到 t<0 交货并翻转 dir=−1 | `t:0→1` @0.0334，→ stage1 | 固定时长（线性 t） | ✅ 30 帧 |

发船初始状态（`DispatchSupplyShip`，行 3762）：`stage=-2, direction=1, t=0`。

### 3.1 固定时长阶段（与几何无关，纯闭式）
这些阶段**速度被强制清零**（`uVel=0,uSpeed=0`），位置由 `t` 的插值给出，时长只取决于步进率：

- 停机盘滑行 / 装卸：`1 / 0.0334 ≈ 30 帧 = 0.5 s`
- 起飞抬升 / 离坞抬升：`1 / num14` 帧
- 降落 / 入坞：`1 / (num14·⅔) = 1.5 / num14` 帧

> **入坞「尾段」时长**（巡航结束→到站交货），去程回程对称：
> `T_dock = 1.5/num14 + 30` 帧。
> a=400 → 323 帧 ≈ **5.38 s**；a=2400 → 173 帧 ≈ **2.88 s**。

### 3.2 巡航阶段（stage 0）——唯一的真·变速运动
99% 的飞行时间在此。逐帧逻辑（行 2212‑2615）拆成两件事：

**(A) 位置积分（每帧都做）**
```
uPos += uVel * (uSpeed/60) + vectorLF6
```
`vectorLF6` 是「跟随目标行星公转」的修正项（仅当目标行星近时非零）。

**(B) 速度/方向控制（每 num23 帧做一次，num23∈{1,10,30}）**
`num23` 是远空间的 LOD 抽帧优化（位置仍每帧积分，仅转向/调速抽帧），**对平均速度无系统性影响**，估时可忽略。

控制律分曲速 / 航行两态，见 §4。

## 4. 巡航阶段的运动学（控制律拆解）

记 `s = num21 =` 飞船到**目标坞接近点**的距离（≈ 到目标塔距离 `d_target`，坞点比塔高约 25–50m，可忽略）。`ds/dt = -uSpeed`。

### 4.1 航行（sail）态：目标速度是「到站距离」的函数
```csharp
num30 = num21/(uSpeed+0.1) * 0.382;
num32 = uSpeed*num30*num10 + 6*num9 + 0.15*num8;   // 目标速度
num32 = min(num32, shipSailSpeed);
// 跟踪：低于目标则 +num33(=num12/60) 加速；高于则 -num13 减速；num13 极大≈瞬时跟踪
```
关键化简：`uSpeed·num30 = uSpeed·s/(uSpeed+0.1)·0.382 ≈ s·0.382`（当 `uSpeed≫0.1`），于是

$$ v_{target}(s) = \min\big(a,\; \underbrace{0.382\,num10}_{k}\cdot s + \underbrace{(6\,num9 + 0.15\,num8)}_{c}\big) $$

即**目标速度只是 s 的线性函数**（与当前速度近乎无关），控制器又能近乎瞬时跟踪它。所以航行态的速度剖面是一个**纯距离函数**：

$$ v(s) = \min(a,\; k\,s + c),\qquad k = 0.382\,num10,\quad c = 6\,num9 + 0.15\,num8 $$

- **巡航段**：`s > s_dec` 时 `v = a`（匀速）。其中 `s_dec = (a-c)/k`。
- **减速段**：`s ≤ s_dec` 时 `v = k·s + c`，是 `ds/dt = -(k s + c)` 的**指数趋近**，一直到 stage0 出口 `s_end = 6·num10`。
- **加速段**（仅当当前速度 `v0 < a` 且远离目标）：以 `num12` 匀加速到 `a`。
  起步贴行星那一小段（中心距 < 1.5·行星半径 ≈ 300m）用更小的 `num11`，但范围仅约百米、且基本被起飞动画覆盖，可并入误差。

各段闭式：
| 段 | 距离区间 | 时间 |
|---|---|---|
| 加速 `v0→a` | `Δs=(a²-v0²)/(2·num12)` | `t = (a-v0)/num12` |
| 匀速巡航 | `s1→s2 (≥s_dec)` | `t = (s1-s2)/a` |
| 指数减速 | `s1→s2 (≤s_dec)` | `t = (1/k)·ln((k·s1+c)/(k·s2+c))` |

> 注意减速段的份量并不小：a=400 时 `s_dec≈1211m`，从 a 减到 `s_end` 约需 **12.5 s**；这是闭式可精确给出、但「按平均速度估」会算错的部分。

### 4.2 曲速（warp）态：指数升速 + 线性刹车
曲速触发（`warpState=0` 时，行 2267‑2274）需同时满足：
`flag(已研曲速)` ∧ `dist²到出发行星 > 25e6 (即 >5000m)` ∧ `s > warpEnableDist/2` ∧ `uSpeed ≥ a` ∧ `有翘曲器(或免费)`。

进入后（行 2276‑2309）：
```
warpCap = min(shipWarpSpeed, 2·D)          // D = 两行星间距 num15
num24   = warpCap * (1001^warpState − 1)/1000   // 当前曲速增速；warpState=1 → num24=warpCap
uSpeed  = a + num24                         // 总速率，峰值 a + warpCap
num28   = num24*0.0449 + 5000 + a*0.25      // 刹车距离阈值
if s < num28:  warpState -= 1/15  (15帧降到0)   // 刹车
else:          warpState += 1/60  (60帧升到1)   // 升速
// 末端限速：若 num24/60 > (s-num28) 则 num24 = (s-num28)*60*1.01
```

**性质**：
- `warpState` 以 `dt` 为自变量线性升/降（升 1/帧·秒、降 4/帧·秒），`num24` 随之**指数**变化；
- 升速段约 **1 s**（`warpState:0→1`），其间从 `a` 指数升到 `a+warpCap`；
- **刹车**：末端限速项 `num24 ≤ 60·(s-num28)` 使 `s` 跌破 `num28` 时 `num24` 立刻被压到 0。
  即**曲速在 `s ≈ s_brake = warpCap·0.0449 + 5000 + 0.25a` 处「硬切」回航速 `a`**，刹车过程仅约 0.1 s。
- 之后是 §4.1 的航行接近（从 `s_brake` 匀速 `a` 到 `s_dec`，再指数减速到 `s_end`）。

> **⚠️ 更正（经第 7 节逐帧仿真验证后修订）**：我最初据此推断「曲速在 `s_brake≈23 km` 处硬切回航速 `a`，其后整段亚光速接近以 `a` 匀速走完、长达数十秒」。**这是错的**。`num28` 阈值会随 `num24`（当前曲速）**动态下移**，刹车走廊里 `s` 与 `num28` 互相追逐，飞船以「高速但衰减」的速度冲过整条走廊（约 0.1–几秒），**而非**以 400 m/s 爬行 23 km。真正以航速 `a` 走完的「最终亚光速段」只有约 `5000 + 0.25a ≈ 5.1 km`。该错误使我下面 §5.3 的曲速程估时严重偏长（实测偏差 +85% ~ +250%，见第 7 节），修正方法见 §7.4。

升速段位移（精确积分，`warpState` 从 `w0` 到 1，`dw/dt=1`）：
$$ D_{su}(w_0)=a(1-w_0)+\frac{warpCap}{1000}\!\left[\frac{1001-1001^{w_0}}{\ln 1001}-(1-w_0)\right],\quad t_{su}=1-w_0 $$
（`w0=0` 时 `D_su = a + 0.14374·warpCap`，`ln1001≈6.9088`。）

## 5. 闭式「这一程剩余时间」算法

### 5.1 输入
| 符号 | 含义 | 来源 |
|---|---|---|
| `v0` | 当前速率 `uSpeed` | 飞船 |
| `w0` | 当前曲速进度 `warpState` | 飞船 |
| `hasWarper` | `warperCnt>0`（或 `warperFree`） | 飞船 |
| `d_target` | 到目标塔距离 → `s0` | 几何 |
| `d_start` | 到起始塔距离 | 几何 |
| `scale, warpDrive` | 科研：`a=400·scale, W=400000·scale, flag=warpDrive` | GameHistory |
| `L_w` | `warpEnableDist`（默认 480000） | 站点设置 |

派生：`D ≈ d_start + d_target`（两行星间距，仅用于 `warpCap=min(W,2D)`）。
预计算 `num8,num9,num10,num12,k,c,s_dec,s_end,num14,T_dock`（§2、§4）。

### 5.2 算法（伪代码，O(1)）
```python
T = T_dock                      # 入坞尾段 = (1.5/num14 + 30) 帧，换算成秒
s = s0 = d_target
warpEngaged = warpDrive and (w0 > 0 or (hasWarper and s0 > L_w/2 and d_start > 5000))

if warpEngaged:
    warpCap = min(W, 2*D);  vmax = a + warpCap
    s_brake = warpCap*0.0449 + 5000 + 0.25*a
    # (1) 若仍在航行且未满速，先匀加速到 a 再起曲速
    if w0 == 0 and v0 < a:
        T += (a - v0)/num12
        s -= (a*a - v0*v0)/(2*num12)
        v0 = a
    # (2) 曲速：升速(w0→1) + 峰速巡航 + 硬刹车，降到 s_brake
    if s > s_brake:
        T += (1 - w0)                      # 升速耗时 (w0≥1 则为 0)
        s_mid = s - D_su(w0)               # 升速后位置
        if s_mid > s_brake:
            T += (s_mid - s_brake)/vmax     # 峰速匀速巡航
        T += ln(1001)/60                    # 刹车尾 ≈0.115s
        s = s_brake
    # (3) 曲速后亚光速接近（航速 a）
    # —— 落到 §5.3 的航行收尾
else:
    # 纯航行：若 v0<a 先加速到 a
    if v0 < a:
        d_acc = (a*a - v0*v0)/(2*num12)
        if s - d_acc > s_dec:               # 长途，能到满速
            T += (a - v0)/num12;  s -= d_acc
        else:                               # 短途三角剖面（少见），见 §5.4
            handle_short_trip()

# §5.3 航行收尾（巡航匀速 + 指数减速），warp/sail 两路在此汇合
if s > s_dec:
    T += (s - s_dec)/a;   s = s_dec
if s > s_end:
    T += (1/k)*ln((k*s + c)/(k*s_end + c))
return T            # 秒；×60 得帧，与游戏 int 对齐时四舍五入
```

### 5.3 数值算例（a=400, scale=1, L_w=480000）
- **同系内 8000m 纯航行**（`s0<L_w/2`，不触发曲速，`v0=a`）：巡航 `(8000−1211)/400 = 17.0 s`；减速 12.5 s；入坞 5.38 s → **≈ 34.9 s**。逐帧仿真真值 = 2092 帧 ≈ 34.9 s，**本式 2092 帧，零误差**（见 §7.3）。
- **曲速程**：⚠️ 本节初稿按「曲速后以 a 匀速走 23 km」估出 ≈79 s，**经仿真证伪**（实际 ≈47.6 s）。曲速程的正确闭式见 **§7.4**（采用游戏对刹车走廊的处理后，误差降到 ≤3%）。

### 5.4 取舍版本（更省算力）
- 丢掉曲速升速积分 `D_su`、刹车尾 0.115 s：误差 < ~1 s/程。
- 丢掉起步加速段（直接认为 `v0=a`）：仅影响刚起飞、`v0` 很低的查询；用「当前速度」输入即可规避。
- 短途三角剖面（§5.2 `handle_short_trip`）：当 `s0 < d_acc + s_dec` 时飞船到不了满速，需解二次方程求峰速。此情形仅出现在极近站点（多半根本不会跨站投送），可直接退化为「`v(s)=min(v_reachable, k s+c)`」近似。

## 6. 哪些结论无法仅凭当前状态闭式得到？影响多大？

| 被丢弃/近似的因素 | 代码位置 | 为何不可闭式 | 影响 |
|---|---|---|---|
| **引力井避让 / 绕行转向**（找最近天体 `num34`，叠加 `vector2/3`、`vectorLF6` 侧推与公转跟随） | 行 2357‑2488 | 取决于沿途**所有**恒星/行星的三维位置，`(d_start,d_target)` 无从得知 | 一般使路径多绕 0–5%；最坏情形（出发/到达需绕过本恒星，或有行星正卡在弦上）可显著增时。**系统性偏短**，是头号误差源 |
| **行星公转**（飞行途中目标坞在动，`uPosNext` 跟随项） | 行 2420,2483 | 需轨道参数+时刻；本算法用快照 `s0,D` | 曲速程（秒级）可忽略；**纯航行长途（分钟级）** 目标移动会带来几个百分点误差 |
| **近行星慢加速** `num11`（贴地 a·0.03） | 行 2342 | 取决于精确离地高度/姿态 | 仅起飞后约百米、约 +3 s；用「当前速度」输入即被吸收 |
| **曲速刹车的自洽 `num28`**（阈值随当前速度动态变化） | 行 2279‑2308 | 隐式耦合 | 用满速 `s_brake` 近似，误差几百 ms |
| **num23 抽帧 LOD** | 行 2311‑2323 | 位置仍每帧积分 | 对平均速度无系统影响，可忽略 |
| **整帧离散与 t 步进取整** | 全程 | 每个阶段边界 ≤1 帧取整 | 合计几帧，可忽略 |

**总体判断**：在「无大天体挡路」的常规直线投送下，本闭式与逐帧模拟的误差**通常在 1–3 秒以内（< 数个百分点）**，主导项（曲速峰速巡航、曲速后亚光速接近、指数减速、入坞尾段）都被精确捕捉。唯一可能产生**较大**偏差的是**引力井绕行**与**长途航行中的行星公转**——二者本质上都不是「当前状态的函数」，是路径/时序依赖量，舍弃它们换来 O(1) 计算是合理的工程取舍。

## 7. 与游戏自带 `CalcRemoteSingleTripTime` / `CalcArrivalRemainingTime` 对比

> 本节于第 1–6 节分析**完成后**才打开这两个方法（行 2962、3047）。

### 7.1 两个方法是什么
- **`CalcRemoteSingleTripTime`**（行 2962）：给定两站，算**整趟单程**（停机盘→目标坞）时间，单位「帧」。它是一个**纯闭式**（无模拟），用于调度/ETA。
- **`CalcArrivalRemainingTime`**（行 3047）：算某艘**在役飞船的剩余到站时间**，单位「帧」——**这正是题目要求的东西**，与本文 §5 一一对应。
  - 非 stage 0（进/出坞动画）：用 `stage*direction` 区分「出坞段」（= 整程 − 已用动画）与「入坞段」（= 仅剩动画帧），直接 `t·30` / `1.5t/num14` 给出，纯闭式。
  - stage 0：本质就是「从当前位置/速度/曲速态出发的 `CalcRemoteSingleTripTime`」。

### 7.2 它们和我的推导「同源」
逐项核对，二者与我 §4/§5 的拆解**结构完全一致**，许多式子**逐字符相同**：

| 量 | 我的记号 | 游戏记号 | 是否一致 |
|---|---|---|---|
| 速度档位系数 | num8/num9/num10 | num/num2/num3 | ✅ 完全相同 |
| 入坞尾段 | `1.5/num14 + 30` | `30 + (int)(1.5/num5+1)` | ✅ 相同 |
| 减速段时间 | `(1/k)·ln((k·s_dec+c)/(k·s_end+c))` | `num14 = (ln(num11·num13/num12+1) − ln(num11·6num3/num12+1))/num11·60` | ✅ **代数恒等** |
| 巡航匀速 | `(s−s_dec)/a` | `(num8−num13)/a·60` | ✅ 相同 |
| 曲速峰速 | `a + warpCap`，`warpCap=min(W,2D)` | `num21+a`，`num21=min(2·mag,W)` | ✅ 相同 |
| 曲速升降速开销 | `D_su+D_sd ≈ 1.25a + 0.1797·warpCap` | `num22 = 0.18776·warpCap + 1.3333a`（外加扁平 80 帧） | ✅ 思路相同、系数微差 |

**结论：我独立逆出的运动模型和官方闭式是同一套数学**——官方版本就是把它**补全所有分支、调好系数**后的成品。

1. **就「这一程剩余时间」而言，官方 `CalcArrivalRemainingTime` 比我（初稿）算得更准**，尤其在**曲速**场景；它是把我这套同源闭式**补全分支 + 校准系数**后的完整体。**纯航行**场景二者并列、均近乎精确。
2. 我的失误集中在**曲速刹车走廊的建模**——这恰恰印证了 §6 的判断：曲速段的动态（`num28` 自洽刹车）是整段逻辑里最难「想当然闭式化」的地方。**验证驱动**（用逐帧仿真当真值反推）是发现该错误的关键手段——而这正是游戏作者写出准确闭式所依赖的同一手段。
3. 三者本质都是**同一逐帧物理的近似**：`CalcArrivalRemainingTime` 与（修正后的）我的式子都**忽略引力井绕行与行星公转**（§6），故在真实 3D 中二者仍有数个百分点的同源系统误差；但在「直线理想」基准下，**官方式子是经过校准的、可信的那一个**。

## 8. 起降几何修正：切向发射的影响（补充分析）

> 起因：原文 §3/§4 把起飞当成"沿径向垂直离场、之后直线奔向目标"。实际上**发射初速是切向的**——本节予以纠正并用忠实 3D 仿真量化。

### 8.1 纠正：发射速度沿切向（垂线的切线方向），不是垂直起飞

从 `Init` 与 stage 机的代码可证（行号见 `StationComponent.cs`）：

```csharp
shipDockRot = entity.rot;                              // 行星面上：up(Y)=径向(法线), X/Z=切平面
shipDiskRot[i] = shipDockRot * Quaternion.Euler(0,yaw,0);  // 仅绕 Y 偏航 → up 仍=径向, forward(Z)=切向
// stage -1 (起飞): 位置沿 shipDiskPos.normalized(≈径向) 抬升 25m, 但
reference2.uRot = reference.uRot * shipDiskRot[shipIndex];  // 朝向不变, forward 仍是切向
// stage 0 第一帧:
reference2.uRot.ForwardUp(out reference2.uVel, out up);     // uVel = forward = 切向!  up = 径向
```

即：**stage −1 抬升的是位置（径向 +25m，且此阶段速度=0）；一旦进入 stage 0，飞船以 `uSpeed` 从 0 起步、速度方向是切向（平行行星表面）**，随后由转向控制律把 `uVel` 逐渐扳向目标坞方向 `vectorLF4/num21`。这带来两个被原直线模型忽略的代价：

1. **滞留近行星慢加速壳**：船以切向飞行会在「中心距 < 1.5·R」的壳层里待更久（径向逃逸则很快穿出），而壳内加速度是 `num11=a·0.03`（远低于 `num12=a·0.12·num10`）→ 加速更慢。
2. **转向损耗**：发射朝向与目标方向有夹角（取决于该船位的偏航与目标在天球上的位置），转向期间速度的「指向目标分量」被 `cosθ` 折损 → 净进度变慢。转角越大、损耗越大。

### 8.2 量化（忠实 3D 仿真，`launch_sim3d.py`）

把 stage 0 的**完整 3D 转向**（四元数 `ForwardUp`、叉乘角速度控制、近天体避让/斥力，行 2322‑2588）逐行移植为 3D 仿真，与「直线径向」基准对比。**自检**：令 R=1（无近行星壳）且发射方向对准目标时，3D 与直线 `Δ=0`，证明差距纯由起降几何产生、非仿真 bug。

**起飞额外耗时（s0=100000m 纯航行，Δ = 3D − 直线）：**

| a (m/s) | 地平·对准(φ90) | 仰角45° | 天顶·径向(φ0) | 目标在身后(yaw180) |
|---|---|---|---|---|
| 400  | +1.4 s | +1.5 s | +2.3 s | **+5.3 s** |
| 1600 | +0.8 s | +1.0 s | +1.6 s | **+4.3 s** |

**着陆额外耗时（进场 s0=3000m，Δ = 3D − 直线减速）：**

| a (m/s) | 正对直入 | 侧向90° | 切向掠过 |
|---|---|---|---|
| 400  | +0.0 s | +0.9 s | +0.8 s |
| 1600 | +0.0 s | +0.3 s | +0.2 s |

**结论**：单程「起降」总附加 ≈ **+1.4 ~ +6 s**（a=400），**几乎全部来自起飞**（着陆只 0~0.9s，因为进场时船速快、提前对准坞点，且减速段本就被 §4.1 闭式覆盖）。附加量随**转角**增大，最坏（目标恰在发射朝向背面）最大。

### 8.3 对三种估时方法的影响

| 方法 | 是否受影响 | 说明 |
|---|---|---|
| **本文/官方 的「中途剩余时间」** | **几乎不受影响** | 中途查询时起飞瞬态早已结束，当前 `(s0,v0)` 已把结果编码进去；§7 的中途快照误差仍 ≤0.03s |
| **官方 `CalcRemoteSingleTripTime`（整程/ETA）** | 受影响但**已大半修正** | 它用 `num7 = sqrt((dir==1?2:0.5)·R·num5)` 建模近行星壳——**`2`(起飞)/`0.5`(着陆) 正是起降不对称**。实测其 stage0 估计 vs 3D 真值：常规几何仅 **−0.6 ~ +0.5 s**，仅在「目标在身后」最坏几何下偏短 **≈3.4 s**（那部分转角它无从得知） |
| **本文闭式（从坞起算）** | 受影响、**未修正** | 我的 §5 漏了 `num7` 近行星项，从静止起算会少算起飞 **≈2.3 ~ 5.3 s**。**改进**：加入 `num7=sqrt(2·R·a·0.03)`，把"0→num7 走慢加速、num7→a 走快加速"两段拆开，可把常规几何误差压到 <1 s（与官方一致） |

> 官方「中途版」`CalcArrivalRemainingTime` 则用 `num9 = sqrt(uSpeed² + (dir==1?8:2)·(1.5R − dist)·num6)`（行 3121），按飞船**当前所处近行星壳层深度**给出有效速度——比整程版更细，这也是它中途误差极小的原因之一。

### 8.4 这部分能否闭式？

- **近行星慢加速壳**部分：**可闭式**——官方 `num7 / num9` 已经做到（一维、`v=√(2·num5·R)` 型），应当补进我的 §5。
- **转向损耗**部分：**不可**仅凭 `(d_start, d_target)` 闭式——它取决于「该船位偏航朝向」与「目标在天球上的方位」之间的 3D 夹角（每船位的偏航近似随机），属 §6 所列**路径/几何相关、非当前状态函数**那一类。其量级 0~5s、典型 1~2s；作为 O(1) 估时的取舍**舍弃它是合理的**，最坏个例（目标恰在背面）会偏短几秒。

### 8.5 小结
原文"垂直起飞→直线奔袭"的图像**不准确**：飞船**切向**离场再转向，单程因此多花 ≈1.4~6 s（几乎全在起飞）。这一项官方整程闭式靠 `num7` 的 `2/0.5` 不对称因子已基本吸收（残差仅最坏几何的几秒），而我的闭式应补 `num7` 近行星段；剩下的纯转角损耗对任何「仅凭当前状态」的闭式都是不可得量，舍弃影响小。**对"中途剩余时间"这一主用例则无实际影响。**
